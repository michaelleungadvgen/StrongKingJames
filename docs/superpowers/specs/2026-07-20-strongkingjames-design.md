# StrongKingJames — Design

**Date:** 2026-07-20
**Status:** Approved by user
**License:** MIT (open source)

## Purpose

An open-source Bible study tool built on the public-domain King James Version tagged
with Strong's numbers. Users browse scripture, tap any word to see its Hebrew/Greek
lexicon entry, search by Strong's number or by meaning, and ask an AI assistant
questions answered from retrieved scripture with verse citations. Everything runs
locally: PostgreSQL + pgvector for storage and vector search, Ollama for embeddings
and chat. No cloud services, no API keys.

## Primary user experience

A **study tool with AI assist** — the verse browser and Strong's concordance are the
core; the RAG chat is a secondary feature.

1. **Browse** — pick a book and chapter, read the KJV text. Every word carrying a
   Strong's tag is interactive.
2. **Word tap** — clicking a tagged word opens a popover with the Strong's entry
   (number, original-language lemma, transliteration, pronunciation, definition,
   KJV usage), plus a link to list every verse using that number.
3. **Search** — three modes in one search box:
   - Reference (e.g. `John 3:16`) — direct navigation.
   - Strong's number (e.g. `H7225`, `G26`) — all verses containing that number.
   - Free text — semantic search via embeddings, ranked verses.
4. **AI chat** — a panel where the user asks natural-language questions. The answer
   is generated by a local Ollama chat model constrained to the retrieved passages,
   with `Book chapter:verse` citations.

## Architecture

**One deployable:** a Blazor Server app that also hosts REST endpoints. A WPF client
is a later phase and will consume those same endpoints (or the shared libraries
directly in a local-DB mode). This was chosen over a separate API project (more
plumbing for no current benefit) and over "no API until WPF" (would require
reworking Blazor pages later).

### Solution layout (.NET 10)

| Project | Responsibility |
|---|---|
| `StrongKingJames.Core` | Domain models and service interfaces: verses, Strong's entries, search, RAG pipeline. No infrastructure dependencies. |
| `StrongKingJames.Data` | EF Core + Npgsql + `pgvector-dotnet`: schema, repositories, vector similarity queries. |
| `StrongKingJames.Importer` | Console app: parses source data, seeds PostgreSQL, batch-generates embeddings via Ollama. Run once at setup. |
| `StrongKingJames.Web` | Blazor Server UI + minimal-API endpoints under `/api/` for future non-web clients. |
| `StrongKingJames.Wpf` | **Phase 2 — out of scope for this design.** Desktop client over the same API/libraries. |

Test projects: `StrongKingJames.Core.Tests`, `StrongKingJames.Data.Tests` (Testcontainers),
`StrongKingJames.Importer.Tests`.

### External services

- **PostgreSQL 16+ with pgvector** — all data and vector search.
- **Ollama** — `nomic-embed-text` (768-dim) for embeddings; a configurable chat model
  (default `llama3.1`) for answers. Reached over HTTP (`http://localhost:11434` by
  default, configurable).

## Source data

- **KJV text with Strong's tags:** openscriptures **KJV OSIS XML** (public domain).
  Word-level `<w lemma="strong:H430">` tagging.
- **Lexicons:** openscriptures Strong's Hebrew and Greek dictionary XML files
  (public domain).

The importer takes local file paths to these files; the README documents where to
download them. They are not committed to the repo (size), but their public-domain
status is documented.

## Data model (PostgreSQL)

- `books` — 66 rows: `id`, `name`, `abbreviation`, `testament`, `sort_order`.
- `verses` — `id`, `book_id`, `chapter`, `verse`, `osis_id` (e.g. `Gen.1.1`), `text`
  (plain KJV text for display and prompting).
- `verse_words` — `id`, `verse_id`, `position`, `word_text`, `strongs_number`
  (nullable; a word may carry none). A word with multiple Strong's tags produces
  multiple rows at the same position. Indexed on `strongs_number` for concordance
  lookups. **The browse view is reconstructed from `verse_words`**, so the importer
  must preserve display fidelity: untagged tokens (punctuation, italicized supplied
  words, plain words) are stored as rows with `strongs_number = NULL`, in order.
  `verses.text` is the concatenated plain text kept for search results and RAG
  prompts, not the source of the interactive rendering.
- `strongs_entries` — `number` (PK, e.g. `H7225`), `lemma`, `transliteration`,
  `pronunciation`, `definition`, `kjv_usage`.
- `verse_embeddings` — `verse_id` (PK/FK), `embedding vector(768)`, with an HNSW
  cosine index.

Embedding granularity is **per verse**. At retrieval time the RAG pipeline expands
each hit with its neighboring verses (±2, clamped to chapter bounds) so the chat
model sees coherent passages rather than isolated verses.

## Key flows

### Import (one-time setup)
1. Parse OSIS XML → books, verses, verse_words.
2. Parse dictionary XML → strongs_entries.
3. Bulk insert (EF Core batching or `COPY`).
4. For each verse, request an embedding from Ollama (batched, resumable — verses
   without an embedding row are the work queue, so re-running continues where it
   left off).

### Semantic search
Query text → Ollama embedding → pgvector cosine top-k (default 20) → ranked verse
list with book/chapter/verse links.

### AI chat (RAG)
1. Embed the question.
2. Retrieve top-k verses (default 8), expand each with neighbors.
3. Build a prompt: system instructions ("answer only from the passages below; cite
   as Book chapter:verse; if the passages don't answer it, say so"), the passages,
   the question.
4. Stream the Ollama chat response into the UI token-by-token.
5. Render citations as links that navigate the browser pane to the verse.

### Error handling
- Ollama unreachable → UI banner explaining Ollama must be running, with the
  configured URL. Only the Ollama-dependent modes are disabled: semantic search and
  chat. Reference navigation, Strong's-number search, browsing, and the concordance
  keep working (they don't need Ollama).
- Model not pulled → surfaced with the exact `ollama pull` command.
- Import failures are per-stage and resumable; partial imports are detectable
  (counts reported at the end).

## API surface (minimal APIs, for future WPF client)

- `GET /api/books`, `GET /api/books/{book}/chapters/{ch}` (verses with word tags)
- `GET /api/strongs/{number}`, `GET /api/strongs/{number}/verses`
- `GET /api/search?q=...&mode=auto|reference|strongs|semantic`
- `POST /api/chat` (streams the RAG answer as Server-Sent Events)

The Blazor UI uses the underlying services directly (same process); the API is a
thin layer over the same service interfaces.

## Testing

- **Unit (xUnit):** OSIS parser, dictionary parser, search-mode detection
  (reference vs Strong's vs free text), RAG prompt assembly, citation extraction.
- **Integration (Testcontainers):** PostgreSQL + pgvector schema, repositories, and
  vector queries against a real database. Ollama is faked behind its service
  interface in tests.
- Embedding/chat calls sit behind `IEmbeddingService` / `IChatService` so all
  pipeline logic is testable without a running model.

## Release / packaging

- GitHub repository, MIT `LICENSE`.
- `docker-compose.yml`: PostgreSQL (pgvector image), Ollama, and the web app.
- README: prerequisites, where to download the OSIS/dictionary XML, `ollama pull`
  commands, importer usage, compose instructions.
- CI (GitHub Actions): build + unit tests on push. The Testcontainers integration
  suite is run locally by contributors, not in CI (deliberate, to keep CI simple
  and Docker-free).

## Out of scope (this phase)

- WPF client (phase 2).
- Other translations beyond KJV (schema doesn't preclude them; not built now).
- Hybrid full-text + vector search (possible future improvement).
- User accounts, notes, highlights.
